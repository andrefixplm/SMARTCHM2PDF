Framework de Integração PLM

SMARTTC2N8N

Conecte os workflows do Teamcenter ao ecossistema de automação do N8N — disparando qualquer processo externo diretamente a partir de etapas do ciclo de vida PLM.

Siemens Teamcenter N8N Automation Self-hosted · Zero licença adicional EPM Workflow Handlers ITK / SOA / REST
O Desafio

Por que o Teamcenter precisa de automação externa?

O Teamcenter gerencia o ciclo de vida do produto com maestria, mas isolar suas automações dentro do PLM limita o potencial de integração com o restante do ecossistema digital da empresa.

⚠ Sem integração (situação atual)

  • Aprovações de workflow não notificam o ERP automaticamente
  • Liberação de documentos exige intervenção humana para gerar PDF
  • Equipes externas ficam sem visibilidade das etapas PLM
  • Integrações customizadas via ITK em C são caras e lentas de desenvolver
  • Cada integração é um projeto independente e não reutilizável
  • Rastreabilidade de processos cruzados é manual e inconsistente

✦ Com SMARTTC2N8N

  • Qualquer etapa do workflow dispara automações N8N em segundos
  • PDF gerado automaticamente ao liberar documentação
  • Notificações instantâneas no Slack, Teams, e-mail e WhatsApp
  • Sync automático com SAP, Jira, Oracle, Salesforce e mais 400 sistemas
  • Framework reutilizável: configure uma vez, replique em qualquer cliente
  • Dashboard de rastreabilidade com histórico completo de automações
Funcionamento

Como a solução opera

O fluxo é disparado por eventos nativos do Teamcenter, sem modificar a lógica de negócio do PLM. O workflow segue normalmente; o N8N cuida do que vem depois.

Flags de Disparo Disponíveis nos Workflows EPM

complete
Task Concluída
Dispara quando o usuário finaliza qualquer task com sucesso. O mais utilizado.
approve
Aprovação de Review
Review Task aprovada formalmente. Ideal para liberar estágios de engenharia.
reject
Rejeição de Review
Review Task rejeitada. Pode acionar criação de tarefas de correção.
start
Task Ativada
Task torna-se ativa no workflow. Útil para notificar responsáveis externos.
perform
Execução da Task
Usuário clica em Perform. Aciona validações externas antes de prosseguir.
abort
Workflow Abortado
Workflow cancelado. Pode reverter ações em sistemas externos.
suspend / resume
Pause e Retomada
Controle de SLA: aciona alertas quando workflow fica parado.
undo
Demotion / Desfazer
Task desfeita (retrocesso). Útil para compensar ações já executadas.
STATUS_CHANGE*
Mudança de Status PLM
Via BMIDE Event — dispara ao mudar Release Status, independente de workflow.
Arquiteturas

4 Formas de Implementar

Cada arquitetura tem seu custo de implementação e nível de robustez. Escolha com base no perfil técnico do cliente e no escopo do projeto.

A
Starter · Mais rápido
Script Bridge (Python / Bash)
O EPM-run-external-command chama um script Python/Bash no servidor TC. O script faz HTTP POST para o webhook N8N com os dados do workflow. Implementação em 1 dia.
EPM-run-external-command -program /opt/tc_bridge/trigger.py -args task=$(task.name) uid=$(target.uid)
Setup em 1 dia Sem compilação C Python fácil de manter
Sem dados PLM ricos Fire-and-forget
B
Produção · Mais robusto
Custom ITK Handler (C/C++)
Handler compilado com o TC toolkit. Acesso completo ao modelo PLM via API ITK: lê atributos, revisões, datasets e relações antes de montar o payload para o N8N.
EPM_ask_name(task_tag, &name); ITEM_ask_id(item_tag, &item_id); // → payload rico para N8N curl_post(n8n_webhook, json_payload);
Payload completo PLM Máxima confiabilidade Suporte a async/sync
Requer build C Setup: 1–2 semanas
C
Enterprise · Desacoplado
BMIDE Events + Bridge Service
Handler registrado no BMIDE para eventos de objeto (SAVE, STATUS_CHANGE). Um serviço intermediário (Java/Python daemon) polled uma fila ou tabela e entrega ao N8N. Cobre cenários sem workflow ativo.
BMIDE Event: ITEM_REVISION → SAVE ↓ Post-Action Handler ↓ Write to JMS / DB queue ↓ Bridge daemon polls ↓ HTTP POST → N8N
Tolerante a falhas Sem workflow obrigatório Alta rastreabilidade
Mais componentes Latência adicional
D
Avançado · Bidirecional
Bidirecional Completo (SOA + Webhook)
Integração de mão-dupla: TC dispara N8N via webhook, e o N8N retorna dados ao TC via SOA REST/SOAP — atualizando atributos, criando objetos ou avançando tasks programaticamente.
TC → N8N: EPM Handler → Webhook N8N → TC: WorkflowService.createInstance() SOA: ItemService.updateProperties() SOA: WorkflowService.performTask()
Loop completo TC recebe dados externos Máxima automação
Maior complexidade Exige SOA habilitado

Possibilidades

Casos de Uso por Setor

Cada combinação de trigger + N8N workflow abre uma automação diferente. Os casos abaixo são prontos para implementação com o framework.

📄
Geração Automática de PDF
Ao liberar uma revisão de documento no workflow, o N8N gera automaticamente o PDF a partir do CHM/DOCX e armazena no servidor ou SharePoint.
complete → gerar PDF
💬
Notificação Slack / Teams
Aprovação de Engineering Change Order notifica o canal #engenharia no Slack com dados do item, revisão, aprovador e link para o AWC.
approve → Slack message
🔄
Sync com SAP / ERP
Quando um BOM (Bill of Materials) é liberado, o N8N sincroniza automaticamente a estrutura de produto com o SAP via RFC ou API REST.
complete → SAP BOM sync
🎫
Criação de Tarefa no Jira
Rejeição de Review cria automaticamente uma tarefa no Jira atribuída ao responsável pelo item, com link para o workflow TC e descrição do problema.
reject → Jira issue
📧
E-mail com Relatório Técnico
Conclusão de etapa de qualidade envia e-mail automático com relatório PDF para lista de distribuição definida por tipo de produto.
complete → e-mail relatório
📊
Atualização de Dashboard
Cada avanço de workflow atualiza automaticamente um dashboard Power BI ou Metabase com KPIs de progresso de engenharia em tempo real.
start → atualizar KPI
🗄️
Backup e Arquivamento
Release de produto aciona arquivamento automático de todos os datasets associados em sistema de gestão documental (SharePoint, S3, Google Drive).
STATUS_CHANGE:Released
✅
Aprovação via Formulário Externo
Aprovadores externos (fornecedores, clientes) recebem link para formulário N8N. A resposta é registrada no TC via SOA e avança o workflow automaticamente.
start → form → approve TC
⚠️
Alerta de SLA Vencido
Workflow suspense por mais de X horas dispara alerta automático para o gestor responsável via Slack, Teams e e-mail, com link direto à task.
suspend → alerta de SLA
Pré-requisitos

O que é necessário para implementar

A solução é desenhada para funcionar com infraestrutura existente de clientes Teamcenter. Não há compra de licença adicional obrigatória.

🏭
Teamcenter (obrigatório)
  • Versão 12.x, 13.x ou 14.x
  • Acesso administrativo ao Workflow Designer
  • Acesso ao servidor de aplicação (para instalar scripts)
  • AWC (Active Workspace) não obrigatório mas recomendado
  • SOA habilitado (para Arquitetura D)
🔀
N8N (obrigatório)
  • N8N >= 1.0 (Community Edition)
  • Docker ou servidor Linux/Windows
  • Mínimo: 2 vCPU, 2 GB RAM
  • Acessível pelo servidor TC (mesma rede ou VPN)
  • HTTPS recomendado (Let's Encrypt ou certificado interno)
🖥️
Servidor TC (para scripts)
  • Python 3.8+ ou Bash instalado
  • Acesso de rede ao servidor N8N
  • Permissão para executar scripts como usuário TC
  • Biblioteca requests (Python) ou curl (Bash)
⚙️
Ambiente de Build (Arq. B)
  • TC toolkit (headers e libs) — mesma versão do servidor
  • GCC / MSVC compatível com versão TC
  • libcurl (para chamadas HTTP no handler)
  • Ambiente BMIDE para registrar handler
  • Acesso de admin ao servidor TC para deploy
🌐
Rede e Segurança
  • Porta do N8N acessível pelo servidor TC (ex.: 5678)
  • Firewall permite saída HTTP/HTTPS do servidor TC
  • Token de autenticação configurado no webhook N8N
  • Opção: proxy reverso Nginx com TLS
  • VPN interna para instalações on-premise
👤
Perfis Necessários
  • Administrador Teamcenter (Workflow Designer)
  • Sysadmin (instalar scripts / Docker)
  • Desenvolvedor PLM (Arq. B — ITK em C)
  • Analista de automação (configurar workflows N8N)
💡
Caminho mínimo viável — implementação em 1 dia
Arquitetura A (Script Bridge) requer apenas: acesso admin ao TC, Python instalado no servidor TC, e um servidor N8N rodando via Docker. Nenhum código C, nenhuma compilação, nenhuma licença adicional. Ideal para PoC com clientes.
